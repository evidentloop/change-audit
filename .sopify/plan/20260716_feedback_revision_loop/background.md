# 变更提案：反馈裁定闭环

## 需求背景

EvidentLoop `v0.1.0a0` 已能在 `audit.html` 中记录人工反馈，但反馈只能留在浏览器或导出为 JSONL，runtime 不会消费它。用户无法把裁定写回当前报告，也无法在同一份报告中继续下一轮反馈。

本方案补齐 code-diff profile 的最小闭环，不扩张到模型复审、自动修复或通用反馈平台。

评分:
- 方案质量: 9/10
- 落地就绪: 9/10

评分理由:
- 优点: 主流程、动作语义、人机职责和报告身份已经确认，并经过独立用户与技术视角复核。
- 扣分: schema `0.3` 是否足以清晰保留模型原判断、人工事件和当前结论，仍需用 fixture 验证。

## 目标与用户收益

```text
在报告中裁定
  → 复制修订指令
  → 粘贴给 AI coding 工具
  → EvidentLoop 更新当前报告
  → 刷新后继续反馈
```

完成后，用户可以：

- 始终操作同一报告目录，不必管理多份默认修订副本。
- 在报告中区分“模型原判断 / 我的裁定 / 当前剩余问题”。
- 修改上一轮裁定，并在报告内部保留可追溯的 revision run。
- 让错误来源、过期反馈和冲突动作明确失败，而不是静默污染报告。
- 仅在明确要求“另存新报告”时，通过 `--out` 保留原报告并生成副本。

JSONL 导出继续作为高级、自动化和故障恢复入口，不作为主要 CTA。

## 变更边界

本方案包含：

1. 反馈解析、身份校验和冲突拒绝。
2. 人工事件持久化、当前结论重算和最小 run lineage。
3. 原目录内安全更新 `audit.json + audit.html`，以及显式另存副本。
4. “复制修订指令”、Skill 定位、隐私提示和失败恢复。

明确不做：

- 模型复审、Prompt Eval、多审查者或自动修改代码。
- JSONL 导入 UI、批量修订或复杂历史界面。
- 跨工作区搜索、通用 workflow/event framework 或预设扩展点。
- 预先升级 schema；只有 fixture 证明 `0.3` 不足时才讨论 `0.4`。

## 影响与风险

- 影响模块: feedback、revision、validation、renderer/JavaScript、CLI/API、Skill、文档与测试，预计约 10–14 个实现文件。
- 身份风险: 用来源 audit SHA-256 与 `graph_id + run_id + target_id + fingerprint` 联合校验，并在提交前复核来源未变化。
- 结论风险: 保留来源模型结论；人工裁定只形成新的当前结论，不冒充模型重新审查。
- 覆盖风险: 候选 JSON/HTML 完整生成并校验后才替换；失败不得把半成品当作成功报告。
- 隐私风险: 复制内容默认不含本地绝对路径，不自动上传反馈。

## 知识库同步

- `project.md`: 记录稳定 `revise` CLI、默认原目录更新和显式 `--out` 语义。
- `blueprint/design.md`: 写入人机分层、反馈重放、来源定位和报告更新契约。
- `blueprint/tasks.md`: 收口最小反馈闭环，保留模型复审和新 artifact profile 等长期任务。
